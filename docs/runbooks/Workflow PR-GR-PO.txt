4. การจัดการสถานะ (Status Tracking Lifecycle)ในการสร้าง Database แนะนำให้ใช้ ENUM Type เพื่อป้องกันการกรอกสถานะผิดเพี้ยน โดยมี Lifecycle ข้ามแอปดังนี้:🟢 ENUM: pr_status (สถานะคำขอซื้อ)'Pending' (รอพิจารณา): สถานะเริ่มต้นเมื่อผู้ขอสร้าง PR'Approved' (เปิดบิลแล้ว): จัดซื้อสร้าง PO สำเร็จ (Data ไหลไปตาราง PO)'Rejected' (ไม่อนุมัติ): จัดซื้อปฏิเสธคำขอ'Bought' (ซื้อให้แล้ว): สถานะพิเศษ ปิดงานทันทีเพราะจัดซื้อทำ Direct PO ให้แล้ว🔵 ENUM: po_status (สถานะใบสั่งซื้อ)'Pending GR' (รอสินค้าเข้า): สถานะเริ่มต้นเมื่อ PO ถูกสร้าง'Draft GR' (กำลังลงสินค้า): เปลี่ยนเมื่อคลังสินค้าเริ่มลงรับของแต่กด "พักบิล" ไว้'Pending Review' (รอตรวจสอบ): คลังลงรับเสร็จแล้ว (แจ้งเตือน LINE ทำงาน) รอผู้ตรวจยืนยัน'GR Completed' (รับสินค้าเสร็จสมบูรณ์): คลังสินค้ายืนยันความถูกต้องเรียบร้อย ข้อมูลเด้งไปหน้า 2-Way Matching ของจัดซื้อ'PO Closed - Ready for APV' (พร้อมทำจ่าย): จัดซื้อกระทบยอด ใส่เลข PO จริงเรียบร้อย รอส่งบัญชี🟠 ENUM: gr_status (สถานะการรับสินค้า)(สถานะของบรรทัด GR จะล้อตามสถานะของ PO ตั้งแต่ Draft GR เป็นต้นไป)'Draft GR' ➔ 'Pending Review' ➔ 'GR Completed'🔄 สรุป Data Lifecycle (State Machine):พนักงานกดส่ง PR ➔ PR.status = 'Pending'จัดซื้อกด Approve PR ➔ PR.status = 'Approved' ➔ สร้าง Row ใน PO ➔ PO.status = 'Pending GR'คลังสินค้าคีย์รับของ กดพักบิล ➔ สร้าง Row ใน GR ➔ GR.status = 'Draft GR' และ PO.status = 'Draft GR'คลังสินค้ากดรับเสร็จ ➔ GR.status = 'Pending Review' และ PO.status = 'Pending Review' (ยิง LINE)ผู้ตรวจสอบกดยืนยัน ➔ GR.status = 'GR Completed' และ PO.status = 'GR Completed'จัดซื้อเทียบยอดเสร็จ (APV) ➔ PO.status = 'PO Closed - Ready for APV' (End of Process)
